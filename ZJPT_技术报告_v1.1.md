## 第一章：当前口径

### 1.1 一句话定义

ZJPT v1.1 第一阶段不是建平台，也不是做撮合，而是先做一个服务真实京东礼品店的 **投流AI引擎**：用 AI + 人工复核，帮助朋友店完成付费投放分析、计划调整建议、每日复盘和证据留存，验证投流 ROI 是否能真实改善，以及改善能否尽量干净地归因到本引擎的动作。

### 1.2 为什么从投流开始

- 投流是店主已经在花钱的环节，痛感明确。
- 投流 ROI 有后台数据，比分散的全店运营更容易量化。
- 投流动作相对集中，便于记录“做了什么、何时做、数据如何变化”。
- 单店验证成本低，不需要先搭系统，也不需要先组织双边市场。

### 1.3 本阶段不做什么

| 不做 | 原因 |
|---|---|
| 不建完整平台 | 现在最大未知数不是技术能否实现，而是投流改善是否真实存在 |
| 不做泛撮合 | 没有验证出稳定效果前，不引入复杂供需关系 |
| 不做全店代运营 | 客服、库存、选品、供应链会污染归因 |
| 不承诺收益结果 | 只做投流建议、执行辅助和数据复盘，不向店主承诺 ROI 数字 |
| 不碰店主资金流 | 费用与结算按书面约定和第三方支付记录处理，ZJPT 不经手资金 |

| 角色 | 他是谁 | 本阶段做什么 | 产出 |
|---|---|---|---|

## 第三章：第一阶段要完成的 7 件事

### 第1件事：店铺基线采集

目标：知道店铺当前投流水平，避免把自然波动误判成 AI 效果。

至少采集：
- 近 14-30 天广告花费、点击、转化、ROI。
- 主推商品、客单价、毛利区间、库存状态。
- 近期活动节点、平台流量变化、价格调整记录。
- 当前计划结构、关键词、出价、人群、创意素材。

### 第2件事：权限边界确认

目标：确保执行人只能做约定范围内的动作，避免责任不清。

需要写清：
- 哪些后台可看。
- 哪些计划可调。
- 单日预算上限。
- 哪些商品、价格、库存、客服动作不由本引擎负责。
- 店主临时改价、断货、客服异常、平台活动变化如何标记。

### 第3件事：投流诊断

目标：先判断这家店是否值得进入投流实验。

诊断结论分三类：
- **A类**：可试。数据完整，预算足够，商品与库存能承接流量。
- **B类**：先补基础。数据不完整或店铺基础不稳，先做固定范围诊断。
- **C类**：暂停。投流无法解决核心问题，继续投入会污染样本。

### 第4件事：实验计划生成

目标：把每次调整变成可追踪实验，而不是随手改后台。

每个实验记录：
- 实验名称。
- 调整前数据。
- 调整动作。
- 预期影响。
- 生效时间。
- 观察窗口。
- 停止条件。
- 异常标记。

### 第5件事：每日数据留痕

目标：形成可复盘证据链。

每日固定记录：
- 广告花费、点击、转化、成交、ROI。
- 当日执行动作。
- 店主侧异常：价格、库存、客服、活动、物流。
- AI 建议与人工采纳情况。
- 店主确认或补充说明。

### 第6件事：周度复盘

目标：判断是有效改善、随机波动，还是外部因素导致。

周报至少回答：
- ROI 是否改善。
- 改善来自哪些计划或商品。
- 哪些动作可能有效。
- 哪些变化不能归因到本引擎。
- 下周继续、停止或改方向。

### 第7件事：样本沉淀

目标：为后续专家工具化和多店复制打底。

每轮实验结束后沉淀：
- 店铺基线。
- 实验动作。
- 数据变化。
- 人工判断。
- 失败原因。
- 可复用规则。
- 不可复用限制。

## 第四章：现阶段工具

| 任务 | 工具 | 说明 |
|---|---|---|
| 数据记录 | 飞书多维表格 / 腾讯文档 | 低成本、可协作、可留时间戳 |
| 日常沟通 | 微信群 | 店主、执行人、记录人同群 |
| 文档沉淀 | Word / 飞书文档 | 生成诊断、日报、周报和阶段总结 |
| AI 辅助 | Claude / Kimi / 通义千问 | 做数据摘要、异常提示、建议初稿 |
| 数据计算 | Excel / 飞书表格公式 | 先用表格完成，不写系统 |
| 证据包 | 本地文件夹 + 表格索引 | 截图、日报、周报、确认记录统一归档 |

> 第一阶段原则：不写系统，不买重工具，不做自动化幻想。能用表格和文档跑通的，不先开发。

## 第五章：完整数据流

| 步骤 | 操作人 | 产生数据 | 存放位置 |
|---|---|---|---|
| 1. 店铺授权基础数据 | 店主 | 历史投流截图、商品信息、预算信息 | 飞书表格：店铺基线 |
| 2. 建立基线 | 执行人 + AI | 近 14-30 天投流基线 | 飞书表格：基线页 |
| 3. 生成诊断 | AI 初稿，人工复核 | 诊断报告 | 飞书文档 |
| 4. 确认实验边界 | 店主 + 执行人 | 权限、预算、排除项 | 飞书文档 + 微信确认 |
| 5. 每日执行 | 执行人 | 调整动作、后台截图 | 飞书表格：每日记录 |
| 6. 每日复盘 | AI + 执行人 | 日报、异常标记 | 飞书文档 |
| 7. 店主确认 | 店主 | 反馈、异常补充 | 微信群 + 表格索引 |
| 8. 周度复盘 | 执行人 | 周报、归因判断、下周计划 | 飞书文档 |
| 9. 阶段总结 | ZJPT | 是否继续、是否复制、失败原因 | 本仓或本地归档 |

## 第六章：AI 能做什么，不能做什么

### AI 可以做

- 汇总投流数据，生成日报初稿。
- 对比前后 ROI、点击率、转化率、花费结构。
- 提醒异常波动，如花费突增、转化骤降、计划失控。
- 给出调整建议初稿，如预算分配、关键词清理、低效计划暂停。
- 生成周报和阶段总结初稿。
- 把多日动作和结果整理成可复盘记录。

### AI 不能做

- 不能承诺 ROI 结果。
- 不能独立决定花钱。
- 不能替代人判断商品、活动、客服、库存等业务上下文。
- 不能独立处理争议。
- 不能接触或管理资金流。

判断原则：AI 负责“看数、出初稿、减少重复劳动”，人负责“确认、执行、停损、复盘结论”。

## 第七章：本阶段核心指标

| 指标 | 定义 | 如何测量 | 当前状态 |
|---|---|---|---|
| 投流 ROI 改善 | 朋友店投流 ROI 是否较基线改善 | 与 14-30 天基线比较，并标注异常因素 | 0 数据 |
| 归因干净度 | 改善能否较清楚地关联到本引擎动作 | 每个动作有时间戳、观察窗口、排除项 | 0 数据 |
| 店主配合度 | 店主是否按时给数据、确认异常、执行边界 | 每日记录完成率 | 0 数据 |
| 执行成本 | 完成一次日报、一次调整、一次周报需要多少人力 | 记录分钟数和人工步骤 | 0 数据 |
| 复制可能性 | 规则是否能迁移到第二家京东礼品店 | 阶段总结后判断 | 0 数据 |

> 这些不是最终承重墙替代品。它们只是投流AI分支的第一批下田指标，待 AB 时再与仓库判断层统一对齐。

## 第八章：主要风险和处理

| 风险 | 影响 | 处理 |
|---|---|---|
| ROI 改善不明显 | 引擎地基不成立 | 停止扩展，复盘失败原因 |
| 改善无法归因 | 无法收费，也无法验真专家 | 缩小实验范围，保留动作时间戳和排除项 |
| 店主不配合 | 数据断裂，复盘失真 | 先做配合度记录，不把断裂样本当结论 |
| 子账号权限不足 | 无法执行或无法看清数据 | 先列权限清单，缺权限则降级为诊断 |
| AI 建议误导 | 造成预算浪费 | 所有建议人工复核，设置单日预算上限 |
| 单店偶然性太强 | 无法判断能否复制 | 第一店只判“有没有信号”，第二店后再谈复制 |
| 成本过高 | 即使有效也难商业化 | 记录每个环节耗时，优先压缩日报和复盘成本 |

## 第九章：14 天启动路线

| 天数 | 行动 | 产出 |
|---|---|---|
| 第1天 | 取得朋友店基础数据和权限边界 | 基线数据清单 |
| 第2天 | 建立 14-30 天投流基线 | 基线表 |
| 第3天 | 生成第一版诊断报告 | 诊断文档 |
| 第4天 | 和店主确认实验范围、预算上限、排除项 | 实验边界文档 |
| 第5-7天 | 执行第一轮小调整，每天留痕 | 3 天日报 |
| 第8天 | 做第一次短复盘 | 短复盘文档 |
| 第9-12天 | 执行第二轮调整，保留异常标记 | 4 天日报 |
| 第13天 | 生成周度复盘 | 周报 |
| 第14天 | 判断是否继续跑满一个月 | 继续 / 暂停 / 改方向 |

14 天后只做一个判断：这家店的投流数据有没有出现值得继续追的改善信号。不要在这一步谈平台、专家网络或规模化。

## 第十章：信号成立后再考虑的轻量系统

只有当朋友店跑出连续改善信号，并且归因记录基本站得住，才考虑把表格流程做成轻量工具。

| 模块 | 解决问题 | 技术复杂度 |
|---|---|---|
| 数据导入表 | 减少手工录入 | 低 |
| 日报生成器 | 自动生成日报初稿 | 低 |
| 实验记录器 | 记录动作、时间、观察窗口 | 低 |
| 异常标记器 | 标记库存、价格、活动等外部变量 | 中 |
| 周报生成器 | 汇总一周动作和结果 | 低 |
| 证据包生成器 | 打包截图、日报、周报、确认记录 | 中 |

优先级：实验记录器 > 日报生成器 > 异常标记器 > 周报生成器 > 数据导入表。

## 总结

ZJPT v1.1 的技术重点不是建系统，而是把第一家京东礼品店的投流实验跑清楚：

1. ROI 有没有改善。
2. 改善能不能尽量归因到本引擎动作。
3. 店主是否愿意持续配合。
4. 人工成本是否低到未来能复制。

这四件事没有数据前，后续平台、专家工具化、签约作家网络都只是推演。
