# ZJPT 讨论分支 · 单店全AI运营引擎

## 设想（杨原始设想 + Musk引擎路线收敛）
不做对赌/撮合平台。选"一个店铺"为最小验证单元：
自己做能做的AI轮子 + 外采别人成熟的AI轮子，把这一个店铺的运营动作尽量全部AI化，
做成一台"输入店铺→AI端到端运营→可测结果"的引擎（执行在自己手里=CAND-004使能前提）。

## 为什么这条比"对赌平台"对（已有结论佐证）
- 中国平台对标冷读：货拉拉/滴滴靠"高频可测"撑平台；猪八戒"服务撮合+托管"做十几年没大、死于飞单；真正做大的拼多多/美团靠"重做成本结构/拥有底层能力"，不靠撮合费。
- 推论：ZJPT 走平台本体做不大（飞单+低频+无硬护城河）；走"拥有一台别人造不出的AI运营引擎"才可能横向复制做大。
- 与场景G结论(CAND-004滩头)、Musk镜结论同向，三路独立撞同一落点。

## 待推演的关键问题（下次AB讨论用，先不展开成方向）
1. 选哪个店铺/品类做第一台引擎？判据=结果客观快测 + 归因干净 + 现按人力/月费在做 + 高频（不踩戴一下低频墙）。
2. 哪些运营动作能AI接管(自研轮子)、哪些外采现成轮子(不自研)？列"自研 vs 外采"清单。
3. 单位经济：每完成一个达标运营结果的边际成本，是否随单量下降(Musk式成本曲线=护城河)？
4. 收费：基础服务费+净增量分红(归因关用冻结基线/续费关绑系统不绑技巧，见SK资产4"分红双关律")。
5. 真实验证对象：包包那个店是不是第一台引擎的试验场？(承接包包MVP，不另起)

## 纪律
- 不在没跑通一单前谈"做平台/做大"(5-19推演永动机防护)。
- 启动人数不是判据(2-5人只是最小启动单元，见CLAUDE系统指令2026-06-08修订)。
